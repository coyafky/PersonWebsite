# Learning Section Design

Date: 2026-06-19
Status: Approved design, pending implementation plan

## Goal

Add a new `Learning` section to the personal website that organizes study notes by topic. The first topic is `Hermes Learning`, covering ~20 articles from the owner's Obsidian vault (`/Users/fkycoya/Documents/CoyaPersonal/03-Areas/Hermes`). The section is designed to scale to additional topics in the future (e.g. Python, Next.js) without code changes.

The Learning section differs from Blog: Blog holds polished essays and opinions, while Learning holds structured study notes organized by topic with a topic-level introduction page.

## Confirmed Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Route structure | Multi-topic grouping | `/learning/<topic>/<slug>` scales to future topics without adding top-level nav items |
| Frontmatter | Generated by assistant | Hermes articles are plain Markdown with no frontmatter; assistant generates valid metadata for each |
| Topic home page | Introduction + article list | Topic page shows a topic intro (Markdown) followed by article cards |
| Article ordering | By date (newest first) | Consistent with blog/weekly; no extra ordering field needed |
| Content directory | Subdirectory per topic | `content/learning/hermes/` mirrors route structure; reader extended to support subdirectories |

## Schema And Data Model

### Learning Article Schema

Added to `lib/content/schemas.ts`:

```typescript
export const learningSchema = baseContentSchema.extend({
  kind: z.literal("learning"),
  topic: z.string(),
  tags: stringArraySchema,
  lang: z.string().default("zh"),
  updated: z.string().optional(),
});
```

The `topic` field links an article to its topic (e.g. `"hermes"`). Files whose name starts with `_` (e.g. `_index.md`) are treated as topic intro pages and excluded from article lists. This is a filename convention, not a frontmatter field, so the schema stays clean.

`schemaByKind` and `ContentKind` are extended to include `"learning"`. The `SiteContent` union type includes `LearningPost`.

## Reader Changes

`lib/content/reader.ts` gains four new functions. Existing functions are unchanged.

### 1. `readCollection` extension

Add an optional `subDirectory` parameter so the same collection reader can scan a subdirectory:

```typescript
async function readCollection(kind: ContentKind, subDirectory?: string): Promise<SiteContent[]>
```

When `subDirectory` is provided, the directory becomes `content/<kind>/<subDirectory>/`. This is backward compatible: existing calls without `subDirectory` behave exactly as before.

### 2. New exported functions

```typescript
// Articles for a topic, excluding _index.md (files starting with _)
export async function getLearningPosts(topic: string): Promise<LearningPost[]>

// Topic intro file (_index.md)
export async function getLearningTopicIndex(topic: string): Promise<LearningPost | null>

// All topics (for /learning section home auto-discovery)
export async function getLearningTopics(): Promise<TopicSummary[]>

// Single article by topic + slug (for article detail page)
export async function getLearningPostBySlug(topic: string, slug: string): Promise<LearningPost | null>
```

`TopicSummary` type:

```typescript
type TopicSummary = {
  topic: string;       // directory name, e.g. "hermes"
  title: string;       // from _index.md frontmatter
  summary: string;     // from _index.md frontmatter
  articleCount: number;
};
```

`getLearningTopics` scans subdirectories under `content/learning/`, reads each `_index.md`, and returns topic metadata. This is what makes the section scale without code changes.

### 3. `CollectionMap` extension

The `CollectionMap` type is extended to include `learning: LearningPost`. The existing `getContentBySlug` does not need changes because learning uses `getLearningPostBySlug` (with topic + slug) instead.

## Page Implementation

### `/learning` ‚Äî Section Home

`app/(site)/learning/page.tsx`

- Calls `getLearningTopics()`
- Renders a page header + `ContentCard` for each topic
- Each card links to `/learning/<topic>`
- Reuses existing `ContentCard` component

### `/learning/[topic]` ‚Äî Topic Home

`app/(site)/learning/[topic]/page.tsx`

- `generateStaticParams` scans `content/learning/` subdirectories to pre-render all topic pages
- Calls `getLearningTopicIndex(topic)` for the intro
- Calls `getLearningPosts(topic)` for the article list
- Renders: topic intro via `MdxContent` + article list via `ContentCard`
- 404 if topic directory or `_index.md` does not exist

### `/learning/[topic]/[slug]` ‚Äî Article Detail

`app/(site)/learning/[topic]/[slug]/page.tsx`

- `generateStaticParams` generates `{ topic, slug }` pairs for all learning articles
- Calls `getLearningPostBySlug(topic, slug)`
- Renders: `ArticleLayout` (with TOC) + `MdxContent`
- Same layout as blog detail page

### Frontmatter generation rules

| Field | Source |
|-------|--------|
| `title` | Extracted from article H1; if no H1, inferred from filename |
| `date` | Provided by owner, or estimated from content context |
| `summary` | Distilled from the opening paragraphs (1-2 sentences) |
| `tags` | Inferred from content (e.g. `hermes`, `cli`, `memory`, `config`) |
| `status` | `draft` by default; owner changes to `published` after review |
| `topic` | Fixed `hermes` for all articles in this topic |
| `lang` | `zh` (all articles are Chinese) |
| `englishSummary` | Optional; generated if the article is significant |

## Error Handling

- Invalid frontmatter on `status: published` articles throws (same as existing behavior)
- Invalid frontmatter on `status: draft` articles warns and skips (same as existing)
- Missing `_index.md` for a topic: topic home page 404s
- Nonexistent topic or slug: 404
- Duplicate slugs within a topic: throws `Duplicate learning slug`
