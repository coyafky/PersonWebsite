### 3. 🧠 Skill 设计方法论学习

从两个来源深入学习 Skill 设计方法论：

**来源 1：宝玉《怎么做好一个 Skill》**（微信读书）
- Skill 是**可迭代的模块**——不是一次性脚本，像软件一样需要版本迭代
- Skill **可拆分**：庞大 Skill 拆成「主 Skill + 多个 Step 子 Skill」
  - 主 Skill：定义整体流程、决策路由
  - 子 Skill：每个 Step 的具体执行逻辑
- 拆分好处：降低复杂度、提高可维护性、子 Skill 可独立复用

**来源 2：腾讯技术工程 — Skill 架构文章**
- Skill 设计遵循**高内聚、低耦合**的软件工程原则
- 子 Skill 可被不同主 Skill 组合复用
- 与 Hermes 的 profile 隔离 + sub-profile 概念天然吻合

**实践应用**：在 `ark-seedream-car-preview` skill 中，将 `gen.py`（生图）、`feishu_sender.py`（发送）、`gen_and_send.py`（一体化）拆分为三个独立步骤，便于按需组合。

### 6. 🧠 Claude Code 专家团设计 + Prompt-boost

**专家团设计（dispatch 模式）**
学习了 `/dispatch` 命令的设计思路——将需求分配给不同 sub-agent：

```
用户需求
  → Architect（架构师）：需求澄清 + spec + 任务拆解
  → Coder（实现者）：编码执行
  → Tester（测试者）：质量验证
  → Deployer（部署者）：发布上线
```

每个 sub-agent 掌握不同的 skill、有独立的执行流程。不是「一个 AI 做所有事」，而是「多个 AI 各司其职」。

**Prompt-boost（借鉴阿里QQ提示词优化）**
核心思路：不让 AI 直接从自然语言理解项目，而是先让它**阅读项目文档**，结合代码结构和自然语言描述，生成精确的开发提示词。

流程：
1. 定义 skill → 让它深度扫描仓库（技术栈 / 数据模型 / 组件 / 路由 / 命名约定）
2. 理解用户自然语言意图 → 消歧 → 匹配到项目中的具体文件路径
3. 输出结构化 Markdown Prompt（可直接执行）

已在 Personal Website 项目中实现 `.claude/commands/prompt-boost.md`。

## 💡 学到什么

- **AI 辅助开发的效率是质的飞跃**：LanHui-Web 从零到全栈 SaaS（多租户 + 认证 + i18n + E2E），一周 23 commits。这不只是「写得快」，而是「一个人能撑起一支团队」的能力跃迁。
- **Skill 拆分 = 「整理」思维在 AI 层的落地**：W-18 确立的「整理让 AI 能直接调用」和本周的「Skill 可拆分为主 + 子」是同一个思维在不同层面的应用。整理让信息可被 AI 调用，Skill 拆分让流程可被 AI 组合。
- **专家团设计 = 角色分工思维**：Architect 设计不写代码、Coder 实现不越界、Tester 测试不修 bug——与企业管理的职责分离原则一致。每个角色专注于自己的 skill，而不是一个 AI 试图做所有事。
- **Prompt-boost = 上下文工程**：不是让 AI 猜你的项目，而是先给它完整的项目快照（CLAUDE.md + 项目结构 + 命名约定），再让它翻译自然语言。这是从「AI 工具」到「AI 协作者」的范式转变。
